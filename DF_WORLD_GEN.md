## 1. Pipeline Overview

DF world gen is a **deterministic, seed-driven, multi-pass pipeline** where each pass produces a result that is handed down as a constraint to the next. No pass reads state generated by a later pass. All passes are pure functions of `(seed, inputs)` → `output`.

```
[Seed]
  │
  ▼
Pass 1 ─── WorldLayerGenerator ─────────────────► GeneratedWorldMap
  │          (elevation, biomes, geology,             (24×24 tiles)
  │           temperature, moisture, drainage,
  │           world-scale rivers)
  │
  ▼
Pass 2 ─── HistorySimulator ─────────────────────► GeneratedWorldHistory
  │          (civilization simulation, territory,       (factions, sites,
  │           war, sites, roads, historical figures)     roads, legends)
  │
  ▼
Pass 3 ─── RegionLayerGenerator ─────────────────► GeneratedRegionMap
  │          (micro-terrain, biome variant,              (16×16 cells per
  │           region-scale rivers, lakes,                 world tile)
  │           settlements, roads, slope)
  │
  ▼
Pass 4 ─── LocalLayerGenerator ──────────────────► GeneratedEmbarkMap
             (surface heightmap, soil, rock strata,    (48×48×8+ tiles)
              minerals, trees, water, caves, magma,
              creature spawn table)
```

Each generator is **stateless**. The same inputs always produce the same output. This means any region can be regenerated on demand without being stored in memory globally — the DF approach.

## 3. Phase A — World-Scale River Network

### Problem

Rivers currently only exist at the region layer. The `DrainageBand` scalar drives whether a region has water, but there is no connected river graph on the world map. This means:
- No mouth-to-source river paths visible at world zoom
- No concept of which world tiles share a river system
- The `HasRiver` flags on neighboring region tiles are independent noise decisions

### Algorithm: Flow Accumulation on World Heightmap

This is the standard GIS approach (D8 flow direction):

1. For each world tile, compute a **flow direction** — the steepest downhill neighbor among the 8 cardinal/diagonal neighbors.
2. Tiles with no lower neighbor are **sinks** (ocean or lake outlet).
3. Compute **flow accumulation** — each tile's accumulation = 1 + sum of all tiles that flow into it (recursively).
4. Tiles whose accumulation exceeds a threshold are **river tiles** — they carry the combined flow of their upstream basin.
5. Mark river mouths where river tiles meet ocean-elevation tiles.

### Key Design Decisions

- **Ocean threshold**: elevation < 0.28f → ocean tile (no feature generation). Currently all tiles generate land; this needs to gate the pipeline.
- **River threshold**: accumulation ≥ `riverThresholdFraction × totalTiles` (e.g. 0.03f × W×H = ~17 tiles for a 24×24 map).
- **River strength**: `(int)(accumulation / threshold)` clamped 1–5 — drives channel width in the region layer.
- The flow direction array and accumulation array are computed inside `WorldLayerGenerator` and stored on `GeneratedWorldMap`.

## 4. Phase B — Cross-Region River Stitching

### Problem

`RegionLayerGenerator.BuildBoundaryRiverPortals` checks whether the neighbor world tile has `DrainageBand >= 0.32f`. If both tiles pass, it uses `ResolveSharedBoundaryOffset` to compute the boundary crossing. **But** the region generator runs independently for each world tile — there is no coordination once both regions are generated. The result is that a river emerges from the south edge of region (5,3) at column 7, but enters the north edge of region (5,4) at column 11.

### Root Cause

`ResolveSharedBoundaryOffset` derives the offset from a hash of the world seed + the world coord + the edge. If the two adjacent tiles each call this with their own `worldCoord`, the hashes differ.

### Fix: Canonical Shared-Edge Keys

The fix is already partially in the code — `ResolveSharedBoundaryOffset` exists. **The bug is whether both sides pass the same canonical key.** The contract must be:

> The North edge of tile (X, Y) and the South edge of tile (X, Y-1) **are the same physical edge**.  
> Both must hash using the same canonical key: `min((X,Y-1), (X,Y))` in row-major order.

```csharp
// In RegionLayerGenerator:
private static int ResolveSharedBoundaryOffset(int worldSeed, WorldCoord coord, BoundaryEdge edge, int regionDim)
{
    // Canonical key: always express edge from the tile with the lower row-major index.
    var (canonX, canonY, canonEdge) = edge switch
    {
        BoundaryEdge.North => (coord.X, coord.Y - 1, BoundaryEdge.South),  // "south of the tile above"
        BoundaryEdge.West  => (coord.X - 1, coord.Y, BoundaryEdge.East),   // "east of the tile left"
        BoundaryEdge.South => (coord.X, coord.Y, BoundaryEdge.South),
        _                  => (coord.X, coord.Y, BoundaryEdge.East),
    };
    var hash = SeedHash.Hash(worldSeed, canonX, canonY, (int)canonEdge + 1000);
    return Math.Abs(hash) % regionDim;
}
```

This makes the shared boundary deterministic and symmetric. No new files needed — this is a targeted fix inside `RegionLayerGenerator`.

### Validation

- Generate two adjacent world tiles. Extract their region boundary at the shared edge. Assert the boundary crossing column/row matches on both sides.
- Add as a regression test in `RegionLayerGeneratorTests`.

## 5. Phase C — Geological Strata

### Goal

Replace the flat `FillUnderground(map)` (all granite wall) with ordered rock layers whose thickness and type are derived from the world tile's `GeologyProfileId`.

### Concept

DF underground looks like this (top to bottom):
```
Z0  ── Surface (soil/stone)
Z1  ── Subsoil / shallow sediment
Z2  ── Rock layer 1 (primary rock type of geology profile)
Z3  ── Rock layer 2 (secondary rock type)
Z4  ── Rock layer 3 (deeper intrusion)
Z5  ── Cave 1 (Phase F)
Z6  ── Rock layer 4
Z7  ── Cave 2 (Phase F)
...
Zn  ── Magma sea (Phase G)
```

### Data Model

```csharp

    public float AquiferDepthFraction { get; init; }  // 0–1; which depth fraction triggers aquifer

```

### Strata by Geology Profile

Define these in a data class `StrataProfiles` (constants — no hardcoded strings in calling code):

| Profile | Layer 0 | Layer 1 | Layer 2 | Layer 3 |
|---------|---------|---------|---------|---------|
| MixedBedrock | sandstone | limestone | granite | basalt |
| IgneousUplift | granite | diorite | gabbro | basalt |
| SedimentaryWetlands | mudstone | shale | limestone | granite |
| AlluvialBasin | siltstone | sandstone | limestone | granite |
| MetamorphicSpine | marble | quartzite | schist | granite |

### Algorithm in EmbarkGenerator

Replace `FillUnderground`:

```csharp
private static void FillUnderground(GeneratedEmbarkMap map, StrataProfile strata, int seed)
{
    var rng = new Random(SeedHash.Hash(seed, 0, 0, 77777));
    var assignments = BuildStrataAssignments(map.Depth, strata, rng);
    
    for (var z = 1; z < map.Depth; z++)
    {
        var rockTypeId = assignments[z];
        for (var x = 0; x < map.Width; x++)
        for (var y = 0; y < map.Height; y++)
            map.SetTile(x, y, z, WallTile(rockTypeId));
    }
}

// BuildStrataAssignments: iterate layers in order, each consumes Rand(min,max) Z-levels,
// repeat the last layer if depth exceeds all layer thicknesses combined.
```

## 6. Phase D — Mineral Vein Placement

### Goal

Place ore veins inside the rock strata after strata are laid. This is what makes underground worth digging.

### Vein Types

DF has four vein shapes:
- **Cluster**: irregular blob — cellular automaton fill starting from a seed point
- **Layer**: horizontal band across the full width, 1–2 tiles thick
- **Vein**: thin diagonal streak, 1 tile wide
- **Scattered**: individual tiles placed at low density

### Data Model

```csharp
// WorldGen/Geology/MineralVein.cs
public enum VeinShape { Cluster, Layer, Vein, Scattered }

public readonly record struct MineralVeinDef(
    string OreId,            // "iron_ore", "copper_ore", etc.
    VeinShape Shape,
    float Frequency,         // expected veins per depth-unit (0.1 = rare, 1.0 = common)
    int MinZ,                // relative to stratum start
    int MaxZ,                // relative to stratum end
    string RequiredRockType, // only place in this rock
    int SizeMin,
    int SizeMax);
```

Define per-`StrataProfile` in `StrataProfileRegistry`:

| Rock Type | Ore | Shape | Frequency |
|-----------|-----|-------|-----------|
| granite | iron_ore | Vein | 0.3 |
| granite | gold_ore | Scattered | 0.05 |
| limestone | copper_ore | Cluster | 0.25 |
| limestone | coal | Layer | 0.4 |
| sandstone | copper_ore | Cluster | 0.15 |
| basalt | iron_ore | Scattered | 0.2 |
| marble | marble_block | Layer | 0.6 |
| shale | coal | Layer | 0.5 |

### New ID Constants

```csharp
public static class OreIds
{
    public const string IronOre    = "iron_ore";
    public const string CopperOre  = "copper_ore";
    public const string GoldOre    = "gold_ore";
    public const string SilverOre  = "silver_ore";
    public const string Coal       = "coal";
    public const string MarbleBlock = "marble_block";
}
```

### Algorithm

```
For each stratum layer [z_start, z_end]:
  For each MineralVeinDef matching that layer's rock type:
    expectedCount = Frequency × (z_end - z_start) × area / 1000
    actualCount   = Poisson(rng, expectedCount)
    For each vein:
      seedX, seedY, seedZ = rng within layer bounds
      PlaceVein(map, shape, seedX, seedY, seedZ, size, OreId)
```

`GeneratedTile` gains a `string? OreId` field — does not change `TileDefId` (still a wall), but marks mineable ore content. The game layer reads this to determine what items are produced when the tile is dug.

> **Important**: The `GeneratedTile` record is already shared with `GameLogic` via the local embark path. Adding `OreId` is an additive, non-breaking change.

### New Files

```
WorldGen/Geology/
  MineralVeinDef.cs
  VeinPlacer.cs      — static, pure function of (map, veinDef, seed, strataAssignments)
```

## 7. Phase E — Aquifer Layer

### Goal

A specific stratum layer becomes permeable. When a tile in this layer is dug, water seeps into the resulting empty space. This is handled by the game `FluidSystem` at runtime — WorldGen only needs to flag the Z-level range.

### Data Model Change

`StrataProfile` already has `AquiferDepthFraction`. This fraction maps to a specific Z-level band. `EmbarkGenerator` marks those tiles with a new `TileFlag`:

```csharp
// Extend GeneratedTile:
public readonly record struct GeneratedTile(
    string TileDefId,
    string? MaterialId,
    bool IsPassable,
    GeneratedFluidType FluidType = GeneratedFluidType.None,
    byte FluidLevel = 0,
    string? OreId = null,       // Phase D
    bool IsAquifer = false);    // Phase E ← NEW
```

At runtime, when the `TileSystem` processes a dig action on an `IsAquifer` tile, it raises a `AquiferBreakedEvent`, which `FluidSystem` handles by injecting a persistent water source at that cell.

### No new files needed for WorldGen — only a boolean flag on `GeneratedTile` and a line in `EmbarkGenerator`.

## 8. Phase F — Cave Systems (3 Layers)

### Goal

DF has three underground cavern layers, each a distinct biome with its own flora, fauna, and water bodies. They appear at regularly spaced Z intervals beneath the surface.

### Depth Mapping

For a map with `depth = 16` (post-expansion):

| Z range | Contents |
|---------|----------|
| 0 | Surface |
| 1–3 | Subsoil / upper rock (strata) |
| 4 | **Cave 1** |
| 5–7 | Rock (strata) |
| 8 | **Cave 2** |
| 9–11 | Rock (strata) |
| 12 | **Cave 3** |
| 13–14 | Deep rock |
| 15 | Magma sea |

The `LocalGenerationSettings.Depth` is currently defaulted to 8 — this needs to increase to support caves. The viewer already supports Z-slider depth up to 8; extending it is a UI change only.

### Cave Generation Algorithm: Cellular Automaton

DF uses a technique closely related to **cave cellular automaton**:

1. **Seed**: Fill the cave layer with a mix of wall/floor using `CoherentNoise.DomainWarpedFractal2D` — low values = floor, high = wall.
2. **Smooth** (4–5 passes): If a cell has ≥ 5 wall neighbors (Moore neighborhood), it becomes wall; otherwise floor.
3. **Flood fill**: Keep only the largest connected floor region. Isolated pockets become wall.
4. **Connect**: If there are multiple large regions, connect them with a minimum spanning tree of carved corridors.
5. **Decorate**: Place underground water bodies (cave lakes), stalactites on cavity borders, underground fungi/pillars.

```csharp
// New files:
WorldGen/Caves/
  CaveLayer.cs                — data: enum CaveDepth { Cave1=4, Cave2=8, Cave3=12 }
  CaveLayerGenerator.cs       — static; pure function (seed, z, width, height) → bool[,] floorMask
  CaveDecorator.cs            — places water, stalactites, fungi inside floorMask
  UndergroundBiomeIds.cs      — "cave_1", "cave_2", "cave_3" — keys for creature/flora lookup
```

### Integration with EmbarkGenerator

After strata are laid, `EmbarkGenerator` calls `CaveLayerGenerator` for each cave Z-level and carves the floor mask into the map (setting wall tiles to passable floor with the appropriate cave rock type).

### Key Constraint

Cave 1 may have **underground rivers** connecting to surface rivers through fissures. When the surface has an active `LocalRiverPortal`, and the embark map has caves, a `CaveRiverPortal` designates a vertical drop-point where water falls from the surface into Cave 1.

## 9. Phase G — Magma Sea

### Goal

The bottom Z-level band is filled with magma. Digging into it releases an unstoppable magma flood.

### Algorithm

Simple: the bottom 1–2 Z-levels of the embark map are filled with magma fluid tiles instead of rock wall. The `FluidType = GeneratedFluidType.Magma` and `FluidLevel = 7` (max) are set. Surrounding tiles are `basalt_wall`.

No new files. Add `FillMagmaSea(map, magmaZ)` private method to `EmbarkGenerator`.

### Depth Requirement

Current default depth is 8 (`LocalGenerationSettings`). Magma sea at Z-15 requires depth ≥ 16. The `LocalGenerationSettings.Depth` default does not need to change immediately — magma sea is only active when depth ≥ 12. Gate with `if (settings.Depth >= 12)`.

|-----------------|---------|--------------|

### Biome Classification Update (`ResolveMacroBiomeId`)

```
temp < 0.15 && moisture < 0.30  → IcePlains
temp < 0.25                      → Tundra
temp < 0.40 && moisture > 0.55   → BorealForest
elev < 0.28                      → OceanShallow (if elev < 0.15 → OceanDeep)
moisture > 0.74 && temp > 0.65   → TropicalRainforest

temp > 0.58 && moisture < 0.40   → Savanna
moisture > 0.50 && temp < 0.55   → ConiferForest (was BorealForest-ish)
elev > 0.78                      → Highland
moisture < 0.28                  → WindsweptSteppe
```

### History Simulation Algorithm

The simulation runs for N years (configurable, default 250) in year steps. Each year:

**Year 0 (Initialization)**
1. Place 3–8 civilization seeds on habitable tiles (`IsOcean = false`, `ElevationBand < 0.75f`).
2. Each civ has a preferred biome — 60% chance to seed in a tile matching it.
3. Each civ starts with 1 town at its capital tile.

**Years 1–N (Annual Loop)**

*Territorial expansion* (every 5 years):
- Each civ attempts to expand into unclaimed tiles adjacent to their territory.
- Expansion probability: `0.4 - (slopeModifier) - (oceanBlock)`. Mountains and ocean are barriers.
- If an expansion step hits another civ's tile: conflict (see war).

*War simulation* (trigger when two civs share a border):
- Compute `militarism` score for each civ (derived from seed, modified by events).
- If `militarismA + militarismB > conflictThreshold`: war declared. One random border tile transfers per-year during war.
- War ends after `10 + rng.Next(20)` years or when one civ loses its capital.

*Site placement* (every 20 years):
- Civs with territory ≥ 5 tiles place a new town in a fertile, non-adjacent-to-existing-site cell.
- Civs with long-running wars build fortresses at border cells.
- Abandoned towns become ruins.
- Isolated high-mountain tiles near old civs may become shrines or tombs.

*Road construction* (every 30 years):
- Civs with ≥ 2 sites route roads between them using A* weighted by `1 + slope * 3` (roads prefer flat terrain).
- Each road stores the path as a `List<WorldCoord>`.

*Historical events* (every year):
- Sample from a weighted event table: flood, plague, dragon attack, peace treaty, founding of artifact, death of ruler.
- These are narrative records stored in `HistoricalEvent` and surfaced in-game in Legends mode.

### New Files

```
WorldGen/History/
  GeneratedWorldHistory.cs
  CivilizationRecord.cs
  SiteRecord.cs
  RoadRecord.cs
  HistoricalEvent.cs
  SiteKindIds.cs
  HistorySimulator.cs          — main entry point; IHistorySimulator interface
  TerritoryExpander.cs         — flood-fill with barriers
  RoadRouter.cs                — A* on WorldCoord grid
  ConflictResolver.cs          — war & territory transfer
  SitePlacer.cs                — site selection & placement rules
```

### Integration with RegionLayerGenerator

`RegionLayerGenerator.Generate` gains an optional `GeneratedWorldHistory? history` parameter. When present:
- If `history.GetSiteAt(worldCoord)` returns a site, `HasSettlement = true` is forced for cells near the region center.
- If roads pass through the world tile, cell-level road logic is weighted higher.
- Ruin sites create `HasRuin = true` boolean on `GeneratedRegionTile` (new field).

### Performance Note

250 years × (W × H) territory checks at 24×24 = 576 tiles = ~144,000 operations. This is trivially fast. At 256×256 world size it becomes ~16M — still < 1 second on modern hardware since each operation is cheap arithmetic.

### Biome → Tree Species Mapping

```csharp
// In EmbarkGenerator, replace Tree placement with:
private static string ResolveTreeSpecies(string biomeId, Random rng)
{
    return biomeId switch
    {
        MacroBiomeIds.TemperatePlains   => Pick(rng, [TreeSpeciesIds.Oak, TreeSpeciesIds.Birch]),
        MacroBiomeIds.ConiferForest     => Pick(rng, [TreeSpeciesIds.Pine, TreeSpeciesIds.Spruce]),
        MacroBiomeIds.MistyMarsh        => TreeSpeciesIds.Willow,
        MacroBiomeIds.Highland          => TreeSpeciesIds.DeadTree,
        MacroBiomeIds.WindsweptSteppe   => TreeSpeciesIds.DeadTree,
        MacroBiomeIds.TropicalRainforest => Pick(rng, [TreeSpeciesIds.Palm, TreeSpeciesIds.Baobab]),
        MacroBiomeIds.BorealForest      => TreeSpeciesIds.Spruce,
        _                               => TreeSpeciesIds.Oak,
    };
}
```

- Trait: `{ type: "choppable", materialId: "{species}_wood" }` 
- Specific sprite in `sprites/tiles.json`

## 16. Testing Strategy

Follow the existing pattern in `DwarfFortress.WorldGen.Tests`: one test class per generator.

### New Test Classes to Add

| Test Class | Key assertions |
|-----------|----------------|
| `WorldRiverNetworkTests` | Connected river path from highland to lowland; no cycles; strength proportional to upstream count |
| `RegionRiverContinuityTests` | Adjacent region tiles share the same boundary crossing offset on their shared edge |
| `StrataGeneratorTests` | Profile assignments cover all Z-levels; layer order matches profile definition; no Z skipped |
| `MineralVeinTests` | Veins only appear in their declared rock type; cluster shapes are connected; no out-of-bounds placement |
| `AquiferTests` | Exactly one Z-level band has `IsAquifer = true` for a profile with `AquiferDepthFraction > 0` |
| `CaveLayerGeneratorTests` | Cave floor mask is fully connected (single region); no cave tiles at wrong Z-level |
| `BiomeExpansionTests` | All (temp, moisture) × elevation combinations map to a known `MacroBiomeIds` constant |
| `HistorySimulatorTests` | After N years, every world tile is either unclaimed or owned by a known civ; no out-of-bounds territory; roads connect only known sites |
| `TreeSpeciesTests` | No `TileDefId == "tree"` in the output (only species IDs); biome-specific species are expected |
| `CreatureSpawnTests` | All spawns land on passable, non-fluid tiles; creature IDs are non-empty strings |

### General Rules

- All tests use `new Random(seed: 42)` / fixed seed for reproducibility — never `Random.Shared`.
- Tests assert on structure/invariants, not on exact coordinates (which are seed-dependent and may shift if algorithms improve).
- Tests must not take > 500ms each. Use small map sizes (8×8 world, 8×8 region, 16×16×8 local) for unit tests.
- Use `Xunit` `Theory` + `InlineData` when asserting against multiple biome IDs or geology profiles.
